# RL Video Benchmarking

## Overview

Benchmark the repo's scenario, recording, and packaging decisions against external precedent instead of inventing content taste from scratch.
Favor transferable patterns from successful RL and AI-simulation videos, not imitation of one creator's persona.
Keep the outcome decision-oriented: what to borrow, what to avoid, and what to change next in this repo.

## Workflow

1. Frame the benchmark question.
Classify the user's need as one of:
- `Concept fit`: is this scenario worth building for video?
- `Packaging`: what title, thumbnail, hook, or cold open makes it legible?
- `Structure`: what should the first 10 seconds, main arc, and payoff be?
- `Channel precedent`: which creators are the closest comps and why?

2. Load repo context first.
Read `../../references/project-direction.md` for the project's watchability bias.
Read `../../references/video-standard.md` when the request touches recording, camera, viewer clarity, or cut structure.
Read `references/youtube-patterns.md` when you need the benchmark set, win patterns, or anti-patterns.

3. Map the request to one or two benchmark archetypes.
Use:
- `Viral spectacle`: Code Bullet, SethBling, Pezzza's Work, Jabrils
- `Polished simulation documentary`: Sebastian Lague, Primer
- `Trusted explainer`: sentdex, Two Minute Papers, AI and Games
Explain which archetype matches and which does not.

4. Score viewer value before RL cleverness.
Judge the candidate on:
- instant goal clarity
- visible danger or failure
- visible learning progress
- thumbnailable contrast
- personifiable agent behavior
- one memorable exploit, twist, or reveal
- feasibility in this repo

5. Prefer borrowing structure, not surface style.
Recommend the smallest useful set of tactics to copy:
- title pattern
- thumbnail contrast
- cold-open shot
- escalation ladder
- failure montage
- comparison shot
Do not recommend copying jokes, cadence, or creator persona verbatim.

6. Convert the benchmark into repo action.
When the idea is weak, change the scenario before suggesting better editing.
When the idea is strong but packaging is weak, suggest:
- a clearer title promise
- a simpler first shot
- more readable danger, target, or reset choreography
- a tighter payoff or comparison clip

7. Hand off cleanly.
If the concept itself changes, return to `scenario-ideation` or `scenario-spec`.
If the concept is set and the question is how to record or cut it, hand off to `scenario-record`.
If the benchmark is agreed and the user wants concrete title / thumbnail / cold-open / edit packaging, hand off to `rl-video-packaging`.
If the user instead wants a broad go/no-go quality review on an existing artifact, hand off to `rl-video-quality-gate`.

## Decision Rules

- Prefer verbs and spectacle over algorithm names in titles.
- Prefer one protagonist and one readable problem over many simultaneous systems.
- Prefer visible failure and recovery over hidden score gains.
- Prefer "watch the agent struggle, then suddenly get it" over long incremental charts.
- Treat funny failure, eerie competence, and exploit-seeking as assets, not embarrassments.
- If the benchmark requires invisible reasoning to be impressive, it is a poor fit for this project.

## Response Shape

- Start with the verdict: `Strong fit`, `Promising but reframe`, or `Weak fit for video`.
- Name the closest benchmark archetype and 2-4 relevant channels or videos.
- List what to borrow.
- List what to avoid.
- End with the next repo step: usually `scenario-ideation`, `scenario-spec`, or `scenario-record`.
